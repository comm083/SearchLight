# Vector Search (Semantic Search)

비정형 장면 묘사 데이터를 벡터화하고 FAISS 인덱스를 관리하는 모듈입니다. (개발자 B 관리)

## B. 세부 과제 체크리스트

- [ ] **장면 묘사 데이터셋**: 50건 이상 영상 상황 텍스트 준비.
- [x] **임베딩 파이프라인**: Sentence-Transformers 기반 벡터 변환.
- [x] **FAISS 벡터 스토어**: 인덱스 생성 및 Top-K 검색 로직 구현.

## B 개발 계획 및 방향성 (FAISS 검색 연동)

개발은 크게 4단계로 나누어, 가장 작은 단위의 핵심 기능부터 점진적으로 살을 붙여나가는 방식(Bottom-up)으로 진행합니다.

- [x] **1단계: 임베딩 파이프라인 및 테스트 데이터 준비**
*   **어떻게 (How):** 테스트용 샘플 텍스트 데이터를 준비하고, SentenceTransformers와 같은 사전 학습된 임베딩 모델을 사용하여 텍스트를 고정 길이의 벡터로 변환하는 파이썬 스크립트를 작성합니다.
*   **왜 (Why):** FAISS는 텍스트를 직접 이해하지 못하고 '숫자로 된 벡터 배열' 간의 거리만 계산합니다. 따라서 코어 검색 엔진을 테스트하려면 반드시 텍스트를 벡터로 바꾸는 선행 파이프라인이 확보되어야 합니다.

- [x] **2단계: FAISS 코어 로직(모듈) 개발**
*   **어떻게 (How):** FastAPI와 무관한 순수 파이썬 클래스(예: `FaissSearchEngine`)를 작성합니다. 이 클래스는 FAISS 인덱스(예: `IndexFlatL2`) 생성, 벡터 추가(add), 디스크에 저장 및 로드, 그리고 가장 유사한 K개의 결과를 찾는 검색(search) 기능을 갖춥니다.
*   **왜 (Why):** 비즈니스 로직(검색 엔진)을 웹 프레임워크(FastAPI)와 분리(Decoupling)하면, 나중에 서버를 띄우지 않고도 검색 품질만 따로 테스트하기 매우 용이하며, 유지보수성이 극대화됩니다.

- [x] **3단계: FastAPI 백엔드 API 연동**
*   **어떻게 (How):** 앞서 만든 `FaissSearchEngine`을 FastAPI 애플리케이션에 연결합니다. 클라이언트의 텍스트 쿼리를 받는 API 엔드포인트(예: `POST /api/search`)를 만들고, 쿼리 벡터화 -> FAISS 검색 -> 결과를 JSON 포맷으로 반환하도록 연결합니다.
*   **왜 (Why):** 다른 시스템이나 프론트엔드에서 우리의 검색 엔진을 사용할 수 있도록 표준화된 RESTful 인터페이스를 열어주기 위함입니다. FastAPI를 쓰면 비동기 처리와 자동화된 문서화(Swagger)를 덤으로 얻을 수 있습니다.

- [x] **4단계: 타 역할(A, C)과의 모듈 통합**
*   **어떻게 (How):**
    *   **C (의도 분류):** 사용자의 입력이 들어오면 먼저 C 파트의 KoELECTRA 모듈을 호출하여 의도를 파악하고, '검색' 의도일 때만 FAISS 검색이 실행되도록 라우팅 로직을 추가합니다.
    *   **A (SQLite 이벤트 로깅):** 검색이 한 번 일어날 때마다(또는 에러가 발생할 때마다) A 파트가 구현한 로깅 함수/API를 호출하여 검색 키워드와 반환 시간을 데이터베이스에 기록합니다.
*   **왜 (Why):** 철저하게 각 기능을 모듈화해두었기 때문에, 이 단계에서는 단순히 서로의 함수를 호출하거나 API를 찌르는 방식으로 복잡한 Git 충돌 없이 우아하게 팀 프로젝트를 통합할 수 있습니다.

### 개발 원칙 (Best Practices)

1.  **단일 책임 원칙 (SRP):** 하나의 파일에 임베딩, 검색, API, 로깅을 모두 몰아넣지 않습니다. `embeddings.py`, `vector_store.py`, `main.py(api)` 등으로 역할을 쪼갭니다.
2.  **독립 테스트 우선:** API 서버를 띄우기 전에, 파이썬 터미널에서 스크립트를 실행해 FAISS 인덱싱과 검색이 의도대로 동작하는지 먼저 검증합니다.
3.  **예외 처리 (Error Handling):** 인덱스 파일이 디스크에 없거나, 벡터 차원이 맞지 않는 등 치명적인 에러 상황에 서버가 죽지 않고 적절한 에러 메시지(500 Internal Server Error 등)를 뱉도록 설계합니다.
