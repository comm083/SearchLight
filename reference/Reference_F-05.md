### **F-05. [검색 이력 및 정형 이벤트 로깅]**

- **목적:** 추후 서비스 개선 및 통계 분석을 위해 사용자들이 어떤 검색어를 입력했고 시스템이 어떻게 의도를 판별했는지 데이터베이스에 기록함.
- **입력:** 사용자의 검색어(String), 시스템이 판별한 의도(String), 호출 타임스탬프
- **출력:** SQLite 데이터베이스(`test_events.db`) 내 `search_logs` 테이블의 레코드 1줄 생성

- **정상 흐름 (사용자 관점):**
  1. FastAPI 서버가 검색 요청을 받고 의도 분류를 마친 즉시 DB 저장 함수를 호출한다.
  2. 시스템 시간을 기반으로 타임스탬프를 생성한다.
  3. `INSERT INTO` 쿼리를 실행하여 검색어와 의도를 DB에 안전하게 커밋(Commit)한다.
  4. 이후 에러 없이 다음 단계(FAISS 검색 등)로 자연스럽게 넘어간다.

- **예외/실패 처리:**
  - DB 파일 권한 문제나 락(Lock)이 걸렸을 경우 로깅 실패로 인해 메인 검색 기능이 멈추지 않도록 예외(Try-Catch) 처리만 하고 경고 로그를 남김.

- **완료 기준 (DoD):** 검색 API를 5회 호출한 뒤 SQLite 뷰어 스크립트로 DB를 열었을 때, 5건의 호출 로그가 정확하게 찍혀 있어야 함.
